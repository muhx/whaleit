<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API (unauth) | Registration, login, verification, password reset — untrusted input |
| client→API (auth) | Authenticated requests via JWT cookie or API key |
| API→PostgreSQL | Server-to-DB, trusted (internal network) |
| API→SMTP server | Server-to-external, email delivery |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-auth-01 | S (Spoofing) | POST /auth/login | mitigate | Argon2id password hashing + constant-time comparison |
| T-auth-02 | T (Tampering) | POST /auth/register | mitigate | Zod validation on frontend, server-side validation of email/password format |
| T-auth-03 | R (Repudiation) | API key usage | accept | API keys logged with prefix only (not full key) |
| T-auth-04 | I (Info disclosure) | API keys table | mitigate | Store SHA-256 hash only; full key returned once at creation |
| T-auth-05 | D (DoS) | POST /auth/register | mitigate | Rate limiting via tower_governor (extend existing login rate limit to register) |
| T-auth-06 | E (Elevation) | /api-keys endpoints | mitigate | Extract user_id from JWT claims (not request body); verify ownership before delete |
| T-auth-07 | S (Spoofing) | API key auth | mitigate | SHA-256 hash comparison with constant-time eq; check is_active and expiry |
| T-auth-08 | I (Info disclosure) | /auth/forgot-password | mitigate | Always return 200 regardless of email existence (prevents enumeration) |
| T-auth-09 | T (Tampering) | verification_tokens | mitigate | SHA-256 hash stored, single-use (consume on verify), 1-hour TTL |
| T-auth-10 | E (Elevation) | Unverified user access | mitigate | require_jwt checks email_verified, returns 403 with X-Verification-Required header |
