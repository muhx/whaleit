**Date:** 2026-04-22 **Commits:** 0e05ba42, 7f6bc29c

## What was built

### Backend (Commit 0e05ba42)

- **PostgreSQL migration** — `users`, `verification_tokens`, `api_keys` tables
- **Diesel schema** for new auth tables
- **Core user module** with `UserRepositoryTrait` and `User`/`NewUser`/`ApiKey`

- **PG repository** (`PgUserRepository`) implementing all user/auth storage
- **Email service** — SMTP via lettre, dev mode logs verification URLs to
  console
- **Auth rewrite** — JWT with user ID + email claims, `wfk_` API key auth, email
  verification gating
- **API endpoints** — register, verify-email, forgot-password, reset-password,
  API key CRUD (list, create, revoke)
- **Config** — `WF_MULTI_USER_AUTH` env var with backward compatibility

### Frontend (Commit 7f6bc29c)

- **Login page** — rewritten for email + password with register/forgot links
- **Register page** — email + password + confirm → pending verification flow
- **Verify email page** — token input + URL `?token=` param support
- **Forgot password page** — email input with "sent" confirmation
- **Reset password page** — token from URL + new password + confirm
- **API keys settings page** — create (shows key once), list, revoke
- **Auth context** — extended with register, verify, forgot, reset, user state,
  needsVerification
- **Zod schemas** — registerSchema, forgotPasswordSchema, resetPasswordSchema,
  createApiKeySchema
- **Routes** — /login, /register, /register/pending, /verify-email,
  /forgot-password, /reset-password, /settings/api-keys

## Key Decisions

- Multi-user auth replaces single-password auth (gated by `WF_MULTI_USER_AUTH`
  env var)
- API keys use `wfk_live_` prefix with 32-char random suffix
- Password reset tokens expire after 1 hour
- Email verification required before login when multi-user mode is enabled
- Backward compatible — single-password auth still works when
  `WF_MULTI_USER_AUTH` is not set
