# Quick Task 20260422: Auth System - Context

**Gathered:** 2026-04-22
**Status:** Ready for planning

<domain>
## Task Boundary

Add register, sign-in, forgot password, create API key, and API secret mechanism in the API and frontend.

</domain>

<decisions>
## Implementation Decisions

### Registration Flow
- Email + password registration with email verification required before access
- Standard secure flow: register → verification email → verify → sign in

### Password Recovery
- SMTP email-based reset link
- User clicks link, sets new password via frontend form
- Reset token with expiry (standard 1-hour TTL)

### API Key Scope & Permissions
- API keys have full user access (same permissions as the user who created them)
- Simpler implementation — no permission scoping needed
- API keys used for programmatic access (Tauri desktop client, scripts, etc.)

### Claude's Discretion
- Database schema for users table, API keys table, verification tokens table
- JWT vs session token implementation details
- Frontend UI patterns for auth forms
- Email sending implementation (SMTP config via env vars)

</decisions>

<specifics>
## Specific Ideas

- Server already has JWT auth via `jsonwebtoken` + `argon2` in `apps/server/src/auth.rs`
- Existing `WF_AUTH_PASSWORD_HASH` is single-password auth — needs to become multi-user
- PostgreSQL is the only storage — all auth tables go there
- Frontend needs login/register pages and API key management UI
- Tauri desktop app connects via API using API_HOST, API_KEY, API_SECRET

</specifics>
