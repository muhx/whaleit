## Summary

The server already has a complete single-password auth system (`apps/server/src/auth.rs`) using Argon2id + JWT + HttpOnly cookies. The task is to evolve this into a multi-user system with registration, email verification, password reset, and API keys. The existing infrastructure (HKDF key derivation, JWT encode/decode, cookie management, `require_jwt` middleware) can be heavily reused — the main additions are: user/accounts tables in PostgreSQL, email sending via lettre, and frontend auth flow pages.

**Primary recommendation:** Extend the existing `AuthManager` to support multi-user lookups; add `users`, `api_keys`, and `verification_tokens` tables via a new PG migration; use `lettre` with tokio feature for SMTP; keep the existing cookie-based JWT pattern for web sessions and add API key as an alternative auth method.

## Existing System Analysis

### Current Auth Flow (single-password)
- **Config:** `WF_AUTH_PASSWORD_HASH` env var → Argon2id PHC string stored in `AuthConfig`
- **Login:** `POST /api/v1/auth/login` with `{ password }` → `AuthManager::verify_password()` → JWT issued → `Set-Cookie: wf_session=<jwt>` (HttpOnly, SameSite=Lax)
- **Middleware:** `require_jwt` extracts token from `Authorization: Bearer` or `wf_session` cookie → validates → optional sliding refresh
- **JWT Claims:** `{ sub: "wealthfolio-web", exp, iat }` — single subject, no user ID
- **Key derivation:** `derive_keys(master)` via HKDF-SHA256 → separate JWT signing key + secrets encryption key
- **Status:** `GET /api/v1/auth/status` → `{ requiresPassword: bool }`

### What Must Change for Multi-User
1. `AuthManager::verify_password()` currently compares against a single stored hash → must look up user by email, compare against per-user hash
2. JWT `sub` claim must contain user ID (not `"wealthfolio-web"`)
3. New tables: `users`, `api_keys`, `verification_tokens`
4. New endpoints: register, verify email, forgot password, reset password, API key CRUD
5. API key auth as alternative to JWT in `extract_token()`
6. `AppState.auth: Option<Arc<AuthManager>>` → always present when multi-user mode is enabled

### What Can Be Reused As-Is
- `AuthManager::issue_token()` / `validate_token()` (just change claims)
- `require_jwt` middleware pattern
- `extract_token()` (add API key path)
- Cookie management (`build_session_cookie`, sliding refresh)
- HKDF key derivation
- `AuthGate` + `AuthProvider` React context pattern
- Rate limiting via `tower_governor`

## Database Schema

### New Migration: `20260422000000_auth_users`

```sql
CREATE TABLE users (
    id TEXT PRIMARY KEY,                    -- UUID v7
    email TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,            -- Argon2id PHC string
    display_name TEXT,
    email_verified BOOLEAN NOT NULL DEFAULT FALSE,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE TABLE verification_tokens (
    id TEXT PRIMARY KEY,                    -- UUID v7
    user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash TEXT NOT NULL,               -- SHA-256 of the token sent to user
    token_type TEXT NOT NULL,               -- 'EMAIL_VERIFY' | 'PASSWORD_RESET'
    expires_at TIMESTAMP NOT NULL,
    used_at TIMESTAMP,                      -- NULL until consumed
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE TABLE api_keys (
    id TEXT PRIMARY KEY,                    -- UUID v7
    user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    key_prefix TEXT NOT NULL,               -- First 8 chars for display: "wfk_live_..."
    key_hash TEXT NOT NULL,                 -- SHA-256 of the full API key
    name TEXT NOT NULL,                     -- User-provided label
    last_used_at TIMESTAMP,
    expires_at TIMESTAMP,                   -- NULL = never expires
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_verification_tokens_hash ON verification_tokens(token_hash);
CREATE INDEX idx_api_keys_hash ON api_keys(key_hash);
CREATE INDEX idx_api_keys_user ON api_keys(user_id);
```

## Backend Architecture

### Auth Module Changes (`apps/server/src/auth.rs`)

**Extend `Claims`:**
```rust
pub(crate) struct Claims {
    sub: String,  // user ID (UUID)
    email: String,
    exp: usize,
    iat: usize,
}
```

**Extend `extract_token()` to support API keys:**
```
1. Authorization: Bearer <jwt>      (existing)
2. wf_session cookie                (existing)
3. Authorization: Bearer wfk_...    (NEW — API key prefix)
```
When the token starts with `wfk_`, look up `SHA-256(token)` in `api_keys` table, load user, inject user into request extensions.

**Add user_id to request extensions:**
After auth, insert `AuthenticatedUser { user_id, email }` into `request.extensions_mut()` so handlers can access the current user.

### API Key Pattern

1. **Generate:** `wfk_live_<32 random bytes, base62 encoded>` — ~43 chars total
2. **Store:** `SHA-256(full_key)` in `api_keys.key_hash`, first 8 chars in `key_prefix`
3. **Return to user ONCE:** full key shown only at creation time
4. **Validate:** On each request with `wfk_` prefix, compute SHA-256, look up in DB
5. **Environment variables for Tauri:** `WF_API_KEY` + `WF_API_SECRET` (or just `WF_API_KEY` containing the full key)

```rust
// API key generation
fn generate_api_key() -> String {
    use rand::Rng;
    let random_bytes: [u8; 32] = rand::thread_rng().gen();
    format!("wfk_live_{}", base64::engine::general_purpose::URL_SAFE_NO_PAD.encode(&random_bytes))
}
```

## Common Pitfalls

### 1. API Key Stored in Plaintext
**What goes wrong:** Storing API keys in plaintext means DB compromise leaks all keys.
**How to avoid:** Store only SHA-256 hash; show full key ONCE at creation.

### 2. Verification Token in URL Leaks to Logs
**What goes wrong:** Token in URL query params gets logged by proxies/browser history.
**How to avoid:** Acceptable for email verification/reset (industry standard). Use short TTL (1 hour) and single-use tokens.

### 3. Single-Password Backward Compatibility
**What goes wrong:** Breaking existing single-password deployments.
**How to avoid:** Keep `WF_AUTH_PASSWORD_HASH` as fallback. If set AND multi-user not configured, fall back to legacy single-password mode. If `WF_MULTI_USER=true` (or similar), use database auth.

### 4. Email Verification Bypass
**What goes wrong:** User can access API before verifying email.
**How to avoid:** Check `email_verified` in `require_jwt` middleware. Unverified users get 403 with `X-Verification-Required: true` header.

### 5. Migration Ordering
**What goes wrong:** Adding tables to the consolidated initial migration instead of creating a new one.
**How to avoid:** Create new migration directory `20260422000000_auth_users/` with `up.sql` and `down.sql`.
